# Product Requirements Document

## 1. 文档信息

- 产品名称：Movie Narrator Tool
- 文档类型：PRD
- 文档版本：v1.0
- 文档日期：2026-04-19
- 适用范围：Windows 桌面版
- 对应项目底座：`movie-narrator-tool-main`

## 2. 项目背景

当前影视解说制作流程中，最耗时的环节不是文案和配音，而是从原片中回找解说视频所对应的镜头，再把匹配到的片段整理到剪辑软件时间线中。现有工具通常偏向以下两类：

- 从原视频自动生成解说视频
- 用文案或语义去找素材

但本项目的核心需求不同：用户已经有一条成品解说视频，需要系统自动识别其中的画面来源，尽量恢复到原片时间轴，并导出可继续编辑的剪映草稿。

## 3. 产品目标

### 3.1 总目标

构建一款桌面软件，输入 `原片` 与 `解说成片` 后，可自动完成：

1. 画面匹配
2. 时间轴恢复
3. 片段补全
4. 剪映草稿导出

### 3.2 核心目标

- 运行时尽量无需人工逐段干预
- 自动输出的高置信结果足够稳定
- 输出结果可以直接进入剪映继续编辑
- 为后续“文案找画面”能力预留接口

### 3.3 指标目标

- `exact` 自动命中片段 precision `>= 99%`
- `exact + inferred` 自动输出 false match rate `<= 3%`
- `exact` 中位时间边界误差 `<= 0.8s`
- 导出时间线不留空片段

## 4. 非目标

以下内容不属于当前版本必须交付范围：

- 面向任意素材库的开放域视频搜索
- 多部影片混剪的全自动片源识别
- 完整替代专业 NLE 的编辑能力
- 将 NarratoAI 作为代码底座直接集成
- 保证所有场景达到 100% 全覆盖和 100% 准确

## 5. 用户与使用场景

### 5.1 目标用户

- 影视解说剪辑师
- 短视频工作室
- 内容运营团队
- 自建影视二创流水线的开发者

### 5.2 核心使用场景

#### 场景 A：外部成品识别

用户已有：

- 一部电影或电视剧原片
- 一条剪好的解说视频

用户希望：

- 自动识别解说视频中的画面对应原片哪个时间段
- 直接导出剪映草稿

#### 场景 B：来源可控生产链

用户使用自家工具制作解说视频，希望在生产时记录来源映射，在导出草稿时直接恢复原片时间轴，尽量减少识别误差。

#### 场景 C：后续语义补画面

当部分镜头无法精确匹配原片时，系统可在后续版本中根据文案或语义自动补充替代画面，避免时间线出现空洞。

## 6. 产品方案总览

产品采用双模式架构：

### 6.1 模式一：来源可控模式

适用前提：解说视频由自有流程或本系统生成。

处理方式：

- 在剪辑阶段记录每个片段的来源信息
- 输出视频时同步生成 `mapping.json`
- 导出草稿时直接读取来源映射

目标：

- 接近无损恢复
- 最大化完整性
- 最小化识别误差

### 6.2 模式二：外部成品识别模式

适用前提：用户拿来的是已完成的成品解说视频，没有来源映射信息。

处理方式：

- 对解说视频先做“低干扰代理视频”净化
- 基于视觉检索、时序对齐、几何验真进行高精度匹配
- 对未稳定命中的片段做自动补全

目标：

- 运行时尽量无人干预
- 自动产出完整时间线
- 保持低误报

## 7. 结果分层定义

为保证“自动化”和“完整性”并存，系统输出结果分为三类：

### 7.1 exact

- 含义：高置信精确匹配
- 来源：检索、对齐、几何验证全部通过
- 用途：直接进入草稿

### 7.2 inferred

- 含义：基于前后锚点和局部时间窗自动补全的结果
- 来源：前后片段已稳定命中，中间片段通过时序关系推断补位
- 用途：直接进入草稿，但标记为推断结果

### 7.3 fallback

- 含义：无法从原片稳定恢复时，用语义或替代素材填补
- 来源：后续“文案找画面”能力或替代镜头策略
- 用途：保持草稿完整，不让时间线断裂

## 9. 核心功能需求

### 9.1 项目管理

系统需要支持：

- 新建项目
- 打开历史项目
- 保存项目状态
- 导入/导出项目描述文件
- 记录匹配模式、路径、阈值、导出设置

### 9.2 媒体导入

系统需要支持：

- 导入原片视频
- 导入解说视频
- 校验视频格式、分辨率、时长
- 自动抽取基础元信息

### 9.3 自动匹配

系统需要支持：

- 原片建库
- 解说视频净化
- 自动镜头切分
- 关键帧提取
- 候选片段检索
- 时序对齐
- 多帧几何验真
- 自动补全

### 9.4 结果查看

系统需要支持：

- 列表形式展示所有片段
- 显示片段类型：`exact / inferred / fallback`
- 显示置信度、边界误差、候选数、状态
- 支持原片/解说视频对照预览
- 支持时间轴查看

### 9.5 导出

系统需要支持：

- 导出剪映国内版草稿
- 导出匹配报告
- 导出项目中间数据

### 9.6 设置

系统需要支持：

- 剪映草稿目录配置
- 模型和阈值配置
- GPU/缓存设置
- 导出参数设置

## 11. 业务规则

### 11.1 自动导出规则

- `exact` 默认自动导出
- `inferred` 默认自动导出并打标
- `fallback` 若已生成替代内容则导出，否则留待后续版本处理

### 11.2 完整性规则

- 系统不得因为单段不稳定就让整体时间线断裂
- 无法精确匹配时优先尝试局部补全
- 无法补全时进入语义替代策略

### 11.3 商业合规规则

- NarratoAI 仅作方案和流程参考
- 不直接复制其非商业许可代码进入本项目

## 12. 成功标准

### 12.1 用户视角

- 用户不再需要手工对照原片逐段回找镜头
- 用户可以直接拿到可编辑的剪映草稿
- 用户只需处理极少数复杂异常情况

### 12.2 产品视角

- 高置信输出稳定
- 自动化流程完整
- 性能可接受
- 日志和结果可追溯

### 12.3 工程视角

- 每次模型或阈值改动都可 benchmark 回归
- 输出结构化结果，便于后续文案补画面和替代策略扩展

## 13. 风险与约束

- 极端二创处理会拉低匹配稳定性
- 多片源混剪超出当前主场景
- GPU、模型下载、FAISS 索引、局部特征匹配会影响首轮处理时间
- 目标应定义为“高置信自动输出稳定”，不宜对外承诺“任意场景 100%”

## 15. 本版本结论

本项目的正确方向不是“从零生成解说视频”，而是“对现有解说成片做原片时间轴恢复”。因此产品方案将以 `movie-narrator-tool-main` 为工程底座，补齐高精度匹配引擎和桌面化交付能力，并用双模式架构解决“高精度”和“完整性”之间的矛盾。
